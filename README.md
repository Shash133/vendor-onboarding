# Vendor Onboarding

An AI-assisted vendor onboarding system that ingests a vendor intake form plus
supporting documents (PAN card, GST certificate, cancelled cheque / bank letter),
runs them through a multi-stage validation pipeline, and produces an
**Approved / Pending / Rejected** decision with a human-readable explanation, a
draft vendor communication, a live per-stage timeline, and a complete audit log.

**Stack:** FastAPI (backend) + Streamlit (frontend) + SQLite (storage) +
Gemini 2.5 Flash (AI agents) + local files. Python 3.11+. No auth.

> The system runs **deterministically even without a Gemini API key**. Every AI
> agent has a deterministic fallback (e.g. fuzzy name matching via `rapidfuzz`,
> rule-weight fraud scoring), and the deterministic validation rules are always
> the source of truth for the final decision. A key only enriches the narrative
> (explanations, classification, extraction) — it never changes the verdict on
> the seeded fixtures.

## Architecture

```
Form + documents
      │
      ▼
upload → classify → extract → validate → consistency → risk → decide → (communicate) → persist/audit
```

| Layer | Location | Responsibility |
| --- | --- | --- |
| HTTP API | `backend/` (FastAPI) | Routes for submissions, uploads, workflow run + SSE stream, decisions, audit, dashboard. Routes are thin; no business logic. |
| AI agents | `agents/` (Gemini 2.5 Flash) | 6 agents: classify, extract, consistency, risk, explanation, communication. Each has a deterministic fallback and never blocks the pipeline. |
| Business logic | `services/` | `workflow_engine` (stage orchestration + events), `validation_engine` (28 rules), `scoring` (4 sub-scores + decision gate), `audit_service`, `storage`. |
| Persistence | `database/` (SQLite) | 8 tables; `schema.sql`, `db.py` access helpers, `seed.py` for prior vendors. |
| Frontend | `frontend/` (Streamlit) | 5 pages: dashboard, submission form, submission detail, workflow timeline (live SSE + poll fallback), audit logs. All data via `api_client.py`. |

**Pipeline stages:** `upload → classify → extract → validate → consistency →
risk → decide → (communicate) → persist/audit`. Scoring computes four sub-scores
(completeness, consistency, compliance, fraud_risk); the decision gate maps them
to approved / pending / rejected.

## Setup

From the `vendor_onboarding/` directory:

```bash
# 1. Create and activate a virtual environment
python -m venv .venv
# macOS / Linux:
source .venv/bin/activate
# Windows (PowerShell):
.venv\Scripts\Activate.ps1

# 2. Install dependencies
pip install -r requirements.txt

# 3. Configure environment (optional — runs deterministically without a key)
cp .env.example .env          # Windows: copy .env.example .env
# then edit .env and set GEMINI_API_KEY=... to enable the live AI narrative
```

`.env` variables:

```
GEMINI_API_KEY=            # optional; leave blank to run via deterministic fallbacks
GEMINI_MODEL=gemini-2.5-flash
DB_PATH=./database/app.db
UPLOAD_DIR=./uploads
BACKEND_URL=http://localhost:8000
```

## Run

The system starts with **two commands** (run from `vendor_onboarding/`, in two
terminals):

```bash
# Terminal 1 — backend (FastAPI on :8000)
uvicorn backend.main:app --reload --port 8000
#   or: make run-backend

# Terminal 2 — frontend (Streamlit on :8501)
streamlit run frontend/app.py --server.port 8501
#   or: make run-frontend
```

Then open the Streamlit UI (default http://localhost:8501).

## Tests

```bash
pytest
#   or: make test
```

The suite covers every API route, all 28 validation rules, the scoring/decision
gate, the workflow engine, and all 10 edge cases (`tests/test_edge_cases.py`).

## Demo

Reset to a clean, seeded state (wipes the SQLite db, re-seeds the prior vendors
needed for the duplicate/fraud cases, and re-generates the fixture PDFs + forms
under `fixtures/data/`):

```bash
make demo-reset
# equivalent without make:
#   python -c "import os; p='database/app.db'; os.remove(p) if os.path.exists(p) else None"
#   python -m database.seed
#   python -m fixtures.generate_fixtures
```

`demo-reset` is idempotent — safe to run repeatedly.

**Suggested demo run order: 1 → 3 → 7 → 9** (happy path → wrong document attached
→ bank holder name mismatch → reused PAN with changed bank details), which walks
an approval, a pending, and two distinct rejection paths.

All **10 edge cases** are generated by `fixtures/generate_fixtures.py` and asserted
end-to-end (expected status, expected failing rule, and communication presence) in
`tests/test_edge_cases.py`:

| # | Scenario | Expected | Headline rule |
| --- | --- | --- | --- |
| 1 | Happy path — clean, consistent vendor | approved | — |
| 2 | Missing mandatory document (no bank proof) | pending | MANDATORY_DOCS_PRESENT |
| 3 | Wrong document attached (GST in PAN slot) | pending | DOC_WRONG_ATTACHED |
| 4 | Illegible PAN scan | pending | DOC_LEGIBLE |
| 5 | PAN format invalid | rejected | PAN_FORMAT |
| 6 | GST not derived from PAN | rejected | GST_PAN_LINK |
| 7 | Bank account holder name mismatch | pending | BANK_HOLDER_MATCH |
| 8 | Fuzzy name variant (same legal entity) | approved | — |
| 9 | Reused PAN with changed bank details | rejected | PAN_REUSE_NEW_BANK |
| 10 | Bank account shared with another vendor | rejected | DUP_BANK_ACCT |

## Deploy (Streamlit Community Cloud — single app, both tiers)

Streamlit Community Cloud runs a single process and cannot expose a second
public port for FastAPI. So the Streamlit app **starts the FastAPI backend in a
background thread** on `127.0.0.1:8000` and talks to it over loopback. One
GitHub-connected Streamlit deploy therefore runs both tiers — no separate
backend host required. (`frontend/server.py` + the `_boot_backend()` hook in
`frontend/app.py` handle this; local `make run-backend`/`run-frontend` is
unaffected.)

Steps:

1. Push this repo to GitHub (already done if you used `gh repo create`).
2. Go to <https://share.streamlit.io> and sign in with GitHub.
3. **Create app** → pick this repo/branch, set **Main file path** to
   `frontend/app.py`, and (if the repo root is the parent folder) leave the
   default; dependencies are read from `requirements.txt`.
4. Under **Advanced settings → Secrets**, paste your config in TOML form
   (see `.streamlit/secrets.toml.example`):

   ```toml
   GEMINI_API_KEY = "your-key"
   GEMINI_MODEL = "gemini-2.5-flash"
   ```

5. Deploy. On first load the app boots the backend, runs `init_db()`, and seeds
   the prior vendors for the duplicate/fraud demo cases.

Notes / limitations:

- **SQLite is ephemeral** on the hosted runtime — data resets when the app
  reboots or redeploys. Fine for a demo; use a managed database (e.g. Postgres)
  for anything persistent.
- The app runs **without** a Gemini key (deterministic fallbacks), but the live
  free tier is rate-limited (~5 requests/min for `gemini-2.5-flash`); agents
  fall back automatically when the quota is hit.
- Never commit `.streamlit/secrets.toml` or `.env` — both are gitignored. Set
  secrets in the Streamlit dashboard instead.
